# Tooldeck 1.2 Planning

本文是 Tooldeck 1.2 版本规划入口。1.2 版本聚焦插件开发体验，目标是让 Tooldeck 插件可以在 monorepo 外部开发，并通过发布后的 npm 包完成插件声明、类型生成、项目校验、构建和运行验证。

当前 1.0 / 1.1 已经跑通可信本地插件的纵向切片：Desktop + CLI 可以扫描 manifest、懒激活 Node 插件、运行 command，并写入 SQLite 历史记录。1.2 不改变 TPP v1 的核心协议边界，而是补齐插件作者在外部项目中开发插件所需的工具链。

## 版本主题

```text
Improve Tooldeck plugin authoring experience.
```

1.2 的核心问题不是新增更多插件能力，而是让 commands-only Node 插件具备可复制、可验证、可发布依赖消费的开发路径。

## 目标

- 插件作者可以在 Tooldeck monorepo 外部创建本地插件项目。
- 外部插件项目可以通过 npm 安装并引用 Tooldeck 插件开发相关包。
- 插件可以使用官方 Vite 插件完成构建。
- 插件项目可以通过开发期命令完成类型生成、manifest 校验、构建和产物检查。
- Tooldeck Desktop / CLI 可以扫描外部插件项目的 manifest，并通过已有流程运行插件 command。
- 内置插件按现有规则自动扫描；外部插件通过显式开发参数加入扫描，不引入插件安装或打包机制。
- Workspace build / typecheck 流程具备确定性，不再依赖全局串行执行作为长期方案。
- Contributor 和 plugin author 有基础 README 或文档说明。
- Desktop 改为单实例应用，或修复多实例启动时后续实例白屏问题。

## 非目标

以下内容不进入 1.2：

- 插件市场。
- 远程插件安装。
- 插件安装包格式。
- 插件 registry 与远程插件发现。
- 非可信插件沙箱。
- 完整权限系统。
- WASM 插件运行时。
- MCP / OpenAPI adapter。
- 复杂 view / table / document 插件模板。
- 插件热更新。

## 包职责

1.2 中插件开发相关包按职责分层：

| 包或入口                  | 职责                                                                          | 面向对象       |
| ------------------------- | ----------------------------------------------------------------------------- | -------------- |
| `@tooldeck/sdk-node`      | 插件运行时 SDK，提供 `definePlugin`、`PluginContext`、command/result helper。 | 插件运行时代码 |
| `@tooldeck/plugin-tools`  | 插件开发期工具包，提供生成、校验、构建编排、测试辅助和项目检查能力。          | 插件作者       |
| `@tooldeck/vite-plugin`   | Vite 构建集成，提供 Tooldeck Node 插件的默认构建规则。                        | 插件项目构建   |
| `@tooldeck/create-plugin` | 创建外部插件项目，生成最小 Node + TypeScript + Vite 插件模板。                | 新插件作者     |

`@tooldeck/sdk-node` 保持运行时 SDK 边界，不引入 Vite、模板、文件生成或打包命令。

1.2 以 `@tooldeck/plugin-tools` 为插件开发体验主线，`@tooldeck/vite-plugin` 是其默认构建后端。生成方向采用 manifest-first：

```text
manifest.json -> generated command types
```

1.2 不从插件 runtime command registration 反推 manifest。

## 成功标准

- 可以通过官方创建器生成一个外部插件项目。
- 生成项目可以安装 npm 依赖并运行 `pnpm build`。
- 生成项目可以运行 `pnpm check`，并完成 manifest 与项目结构校验。
- 生成项目使用 `@tooldeck/sdk-node` 编写 command handler。
- 生成项目使用 `@tooldeck/plugin-tools` 生成 command input 类型和 command id 常量。
- 生成项目使用 `@tooldeck/plugin-tools` 完成 `generate -> check -> build -> check --built`。
- 生成项目可以用 Vitest 配合 `@tooldeck/plugin-tools/testing` 测试 command handler。
- 生成项目可以运行 `tooldeck-plugin inspect` 输出插件项目诊断报告。
- 生成项目使用 `@tooldeck/vite-plugin` 构建 Node 插件入口。
- 构建产物中的 `manifest.runtime.entry` 指向可由 `@tooldeck/host-node` 加载的 ESM 文件。
- Tooldeck CLI 可以通过一个或多个 `--plugin-dir` 指向额外本地插件目录并运行示例 command。
- Desktop 可以通过一个或多个 `--plugin-dir` 或 `TOOLDECK_PLUGIN_DIRS` 扫描同一个外部插件目录并运行示例 command。
- Workspace `build` / `typecheck` / `test` 不依赖全局 `concurrency = 1` 才能稳定通过。
- Contributor 和 plugin author 有基础 README 文档说明 1.2 开发路径。
- Desktop 重复启动时保持单实例，不创建第二套窗口、服务、IPC 或 SQLite 连接，并聚焦现有窗口。

## 建议验收路径

```bash
pnpm dlx @tooldeck/create-plugin my-tooldeck-plugin
cd my-tooldeck-plugin
pnpm install
pnpm check
pnpm build
```

然后从 Tooldeck workspace 验证外部插件：

```bash
pnpm --filter @tooldeck/cli dev -- list commands --plugin-dir ../my-tooldeck-plugin
pnpm --filter @tooldeck/cli dev -- run hello.world --plugin-dir ../my-tooldeck-plugin
pnpm --filter @tooldeck/desktop dev -- --plugin-dir ../my-tooldeck-plugin
```

具体 command id 以模板生成结果为准。
