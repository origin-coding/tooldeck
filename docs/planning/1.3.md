# Tooldeck 1.3 Planning

> **Status (2026-07-20): Implemented; release polishing in progress.** The original
> planning text below is retained as a design record. Current implementation boundaries
> and naming are summarized here and in the accepted ADRs.

## Implementation Status

The 1.3 local distribution loop is implemented on `main`:

```text
external plugin build
  -> pack / dist
  -> local .tdplugin install
  -> builtin / installed / external scan
  -> lazy command activation and history
  -> enable / disable
  -> uninstall
  -> retained-data purge
```

Implementation clarifications made after the initial plan:

- The package-format implementation is the public singular package
  `@tooldeck/plugin-package`, not the initially planned private
  `@tooldeck/plugin-packages`.
- The private `@tooldeck/plugin-management-node` application service coordinates catalog
  synchronization, install, enable/disable state, uninstall, purge, rollback, and cleanup
  reporting for both CLI and Desktop.
- Desktop drag-and-drop installation, installed-plugin uninstall, and retained-data purge
  were completed as part of 1.3.
- Uninstall can complete logically while reporting `cleanupPending` when final quarantine
  cleanup fails; it does not restore a potentially partial install as active.
- The detailed planning documents remain historical design records. Final package
  boundaries are recorded in [ADR 0001](../architecture/decisions/0001-tdplugin-zip-container.md)
  and [ADR 0002](../architecture/decisions/0002-node-plugin-management-service.md).

Code-quality and lifecycle test passes are complete. User documentation, AI Agent
documentation, UX/diagnostic review, and release verification are tracked by
[Issue #20](https://github.com/origin-coding/tooldeck/issues/20).

本文是 Tooldeck 1.3 版本规划入口。1.3 版本聚焦本地插件分发闭环，目标是让 1.2 已经支持的外部插件开发路径可以进一步生成本地安装包，并安装到 Tooldeck 管理的用户级 installed plugin directory 中。

当前 1.2 已经完成插件作者工具链、外部插件目录扫描、懒激活和 command run history。1.3 不改变 TPP v1 的核心协议边界，而是在可信本地插件前提下补齐：

```text
外部插件开发
  -> 插件打包
  -> 插件安装
  -> installed plugin scan
  -> lazy activation
  -> command run
  -> SQLite history
```

## 版本主题

```text
Local Plugin Packaging and Install.
```

1.3 的核心问题不是插件市场，也不是不可信运行时，而是让本地可信插件具备可打包、可安装、可卸载、可启用/禁用、可被 CLI 和 Desktop 扫描运行的最小分发闭环。

## 目标

- 插件作者可以通过 `tooldeck-plugin pack` 从已构建插件项目生成 `.tdplugin` 安装包。
- 插件作者可以通过 `tooldeck-plugin dist` 完成构建和打包。
- `.tdplugin` 是 runtime-agnostic 的 Tooldeck 本地插件包容器；1.3 只实现 Node 插件安装校验。
- Tooldeck CLI 默认扫描 builtin 和 installed 插件。
- `--plugin-dir` 继续作为 1.2 定义的 external plugin dir 增量扫描入口，不改变语义。
- CLI 可以安装、卸载、启用、禁用 installed 插件。
- installed 插件 command 可以正常运行，并写入 SQLite command run history。
- 扫描和安装过程不得 import runtime entry，也不得激活插件代码。
- storage 区分 scanned plugin catalog、installed plugin assets 和 plugin id 级用户状态。
- Desktop 可以在插件页提供窄入口拖拽安装 `.tdplugin`，但该能力不阻塞 CLI P0 闭环。

## 非目标

以下内容不进入 1.3 P0：

- 插件市场。
- 远程插件安装。
- 插件签名。
- 不可信插件沙箱。
- 插件依赖解析。
- 插件热更新。
- 多版本插件并存。
- 插件更新/替换流程。
- 完整权限确认弹窗。
- OS 文件关联。
- 双击 `.tdplugin` 自动打开 Tooldeck。
- `tooldeck://` deep link。
- Desktop router。
- 第二实例参数转发。
- MCP / OpenAPI adapter。

## P0 范围

P0 聚焦 CLI 闭环和核心基础设施：

- 定义 `.tdplugin` 包格式。
- 新增 `@tooldeck/plugin-packages`。
- 新增 `tooldeck-plugin pack` 和 `tooldeck-plugin dist`。
- 扩展 scan source 模型为 `builtin | installed | external`。
- 新增 installed plugin directory 默认扫描。
- 新增 `plugin_installs` 和 `plugin_states`。
- CLI 支持 `tooldeck plugin install`。
- CLI 支持 `tooldeck plugin uninstall`。
- CLI `plugin list` 显示来源。
- CLI `run` 可以运行 installed plugin command 并写入 history。

## P1 范围

P1 在 P0 稳定后推进：

- Desktop 插件页拖拽安装 `.tdplugin`。
- Desktop 显示插件来源。
- Desktop 可选支持卸载 installed 插件。
- `tooldeck plugin purge`。
- 更完整的安装错误展示和诊断信息。

## 扫描来源模型

1.3 明确三类扫描来源：

```text
builtin   = Tooldeck 内置插件
installed = 用户安装到 Tooldeck 管理目录的插件
external  = 通过 --plugin-dir 或 TOOLDECK_PLUGIN_DIRS 指定的开发期插件目录
```

默认扫描顺序：

```text
builtin -> installed
```

传入开发期 external plugin dir 后：

```text
builtin -> installed -> external
```

`--plugin-dir` 继续表示增量 external source，不替换 builtin 或 installed。重复 plugin id 或 command id 继续沿用 manifest index 的 duplicate 错误，不在 1.3 引入覆盖规则。

## 包和存储职责

1.3 新增包和 storage 概念：

| 包或表                      | 职责                                                                  |
| --------------------------- | --------------------------------------------------------------------- |
| `@tooldeck/plugin-packages` | `.tdplugin` 包读写、安全校验、package manifest 校验和 digest helper。 |
| `@tooldeck/plugin-tools`    | 插件作者命令：`pack` 和 `dist`。                                      |
| `plugin_installs`           | Tooldeck 管理的 installed plugin 文件资产。                           |
| `plugin_states`             | plugin id 级用户状态，例如 enabled。                                  |
| `plugins`                   | 当前扫描到的 plugin catalog / registry。                              |

`@tooldeck/plugin-packages` 可依赖 `@tooldeck/protocol`，但不依赖 `@tooldeck/runtime-node`、`@tooldeck/host-node`、`@tooldeck/storage`、Electron 或 React。

## 成功标准

- `tooldeck-plugin pack` 可以生成 `<plugin-id>-<version>.tdplugin`。
- `tooldeck-plugin pack --output <file>` 可以覆盖输出文件路径，且不修改包内 metadata。
- `tooldeck-plugin dist` 可以完成构建和打包。
- `.tdplugin` 包含 root `manifest.json` 和 root `tooldeck-package.json`。
- `.tdplugin` 不包含 `node_modules`。
- CLI 默认扫描 builtin 和 installed 插件。
- `--plugin-dir` 仍然作为 external 增量来源加入扫描。
- `tooldeck plugin install ./my-plugin.tdplugin` 安装插件到用户级 installed plugin directory。
- 安装后 `tooldeck plugin list` 能显示插件来源为 installed。
- 安装后 `tooldeck run <command-id>` 能运行 installed command。
- command execution 写入 SQLite command history。
- 禁用插件后 command 仍可见，但运行时被阻止并记录 error history。
- 卸载 installed 插件后 command 不再出现。
- 安装、扫描、卸载过程不会激活插件 runtime code。

## 建议验收路径

从外部插件项目：

```bash
pnpm check
pnpm build
tooldeck-plugin pack
```

从 Tooldeck CLI：

```bash
tooldeck plugin install ./dev.example.my-plugin-0.1.0.tdplugin
tooldeck plugin list
tooldeck run my.command
tooldeck plugin disable dev.example.my-plugin
tooldeck run my.command
tooldeck plugin enable dev.example.my-plugin
tooldeck plugin uninstall dev.example.my-plugin
tooldeck list commands
```

Desktop 拖拽安装作为 P1，在 CLI 闭环稳定后验收。
